## Исправленный баг: «админ периодически превращается в оператора»

**Причина:** В `AuthContext.tsx` при каждом срабатывании `onAuthStateChange` (вход, выход, **обновление токена**, смена пароля и т.д.) делается запрос в `public.users` за профилем. При любой ошибке или таймауте (15 сек) в состояние подставлялся **fallback-пользователь с ролью `operator`**. В БД роль не менялась — «скатывался» только отображаемый профиль.

Типичные сценарии:
- обновление токена (раз в ~час) + временный сбой/медленная сеть → запрос профиля падает → показ оператора;
- долгий ответ БД → таймаут → снова fallback с `operator`.

**Исправление:** При ошибке/таймауте загрузки профиля не перезаписывать пользователя fallback’ом, если в состоянии уже есть пользователь с тем же `session.user.id`. Тогда сохраняется последний успешный профиль (в т.ч. роль `admin`). Fallback с `operator` используется только при первом входе (когда `prev === null`) или при смене пользователя.

Файл: `context/AuthContext.tsx` — в ветках `else` и `catch` используется `setUser((prev) => ...)` с проверкой `prev?.id === session.user.id`.

## Другие замечания и риски

### 1. База данных

- **Триггер `handle_new_user`** (миграция 001): при `ON CONFLICT (id)` обновляются только `email` и `name`, **role не трогается** — корректно, перезаписи роли из auth нет.
- **Backfill** в конце 001: при повторном запуске тот же `ON CONFLICT` — только email/name. Роль в БД не перезаписывается.
- В коде нет обновления `public.users` (в т.ч. role) — роль меняется только вручную в БД или через будущий админ-функционал.

### 2. Безопасность и RLS

- На таблицах (companies, materials, movements, work_logs и т.д.) политики вида «все authenticated» — разграничение оператор/админ (удаление только своих записей, доступ в History) сделано **на фронте**. При утечке anon-ключа злоумышленник может обходить эти ограничения. Для ужесточения стоит добавить RLS по `role` и/или `created_by`.
- `users`: есть `users_update_self` — пользователь может менять свою строку. Если где-то появится форма «профиль» с полем role (без проверки админа), роль можно будет изменить с клиента. Сейчас таких форм нет.

### 3. Надёжность загрузки профиля

- Один запрос к `users` без повторов. При нестабильной сети можно добавить 1–2 retry при ошибке/таймауте перед тем как оставлять предыдущий пользователь или показывать fallback.
- Таймаут 15 сек — большой; при желании можно уменьшить (например до 5–7 сек) и полагаться на сохранение предыдущего профиля при ошибке.

### 4. Консистентность данных

- В `types.ts` и в `createMovement` используются поля `price_per_ton`, `total_value`; в миграции 001 их нет, они добавлены в 002 — схема согласована.
- В `getMovements` и т.д. используются те же поля — всё сходится с миграциями.

### 5. UX

- При первой загрузке страницы (до ответа от `onAuthStateChange`) показывается общий лоадер AuthGuard — ок.
- После исправления бага с ролью админ не будет «мелькать» в оператора при обновлении токена или кратковременных сбоях.

## Рекомендации на будущее

1. **Роль в БД:** Не хранить роль в `auth.users.raw_user_meta_data` и не обновлять её из auth — только в `public.users`. Триггер и backfill уже не трогают role при конфликте — это сохранять.
2. **Админка ролей:** Если появится экран «управление пользователями», менять `public.users.role` только через сервис/функцию с проверкой «текущий пользователь — admin».
3. **Аудит:** В `audit_log` нет записей по таблице `users`. При изменении ролей или профилей имеет смысл добавить триггер на `public.users` для аудита.
4. **Мониторинг:** Логировать (на бэке или в мониторинге) ошибки запроса профиля из `users`, чтобы видеть частоту сбоев и таймаутов.

Итог: основная причина «админ периодически меняется на оператора» — fallback при сбое загрузки профиля; исправление внесено в `AuthContext.tsx`. Остальное — рекомендации по усилению безопасности и стабильности.

## Правки по заявкам клиента (март 2026)

### 1. Движения: способ оплаты нал/безнал
- В форму «Новое движение» добавлен выбор **Способ оплаты**: Нал / Безнал.
- В таблице движений добавлена колонка **Оплата**.
- В БД: миграция `003_payment_and_status.sql` — колонка `movements.payment_method` (`cash` | `cashless`, по умолчанию `cashless`).

### 2. Движения: редактирование
- В таблице и карточках добавлена кнопка **Редактировать** (иконка карандаша).
- Открывается та же модалка с заполненными полями. Редактировать можно: дату, операцию, вес, цену за тонну, погр./разгр., примечание, способ оплаты. Компания, материал, размер, владение при редактировании недоступны (чтобы не менять позицию). Сумма сделки пересчитывается автоматически.
- API: `updateMovement(id, payload)` в `services/supabase.ts`. Триггер баланса в миграции 003 доработан под UPDATE (при изменении веса/операции пересчитывается баланс позиции).

### 3. Расходы: редактирование и статус оплаты
- В таблице и карточках расходов добавлена кнопка **Редактировать**.
- В форму добавлено поле **Статус оплаты**: Оплачено / Не оплачено. В таблице — колонка **Статус** с бейджами.
- В БД: миграция 003 — колонка `expenses.payment_status` (`paid` | `unpaid`, по умолчанию `unpaid`).
- API: `updateExpense(id, updates)` в `services/supabase.ts`.

### 4. Финансы: статусы оплачено/не оплачено
- На странице «Деньги» в блоке «Расходы» добавлена строка: **Оплачено: X ₽ · Не оплачено: Y ₽** (суммы по полю `payment_status`).

### 5. Защита от потери данных в форме («выкинуло из заявки»)
- При закрытии модалки (крестик или клик по затемнённому фону) при наличии несохранённых изменений показывается подтверждение: «Закрыть без сохранения? Несохранённые данные будут потеряны.» Так снижается риск случайно закрыть форму и потерять примечание или другие введённые данные.
- Реализовано на страницах **Движение металла** и **Расходы**.

### Миграция 003
Перед использованием новых полей и редактирования нужно выполнить в Supabase (SQL Editor) скрипт `supabase/migrations/003_payment_and_status.sql`.
